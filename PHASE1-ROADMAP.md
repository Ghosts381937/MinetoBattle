## MinetoBattle 第一階段開發規劃

目標：在不破壞現有核心循環（商店 → 召喚 → 戰鬥 → 掉落 → 交易所）的前提下，讓每一場戰鬥更有變化，並建立基礎的關卡與裝備成長架構。

---

### 一、敵人多樣化與關卡結構

- [ ] **建立敵人資料結構**
  - [ ] 定義敵人基礎屬性：`id`, `name`, `type`, `level`, `maxHp`, `atk`, `def`, `dropsProfile`。
  - [ ] 加入「敵人類型」標籤（例如：`tank`, `assassin`, `vampire`, `shielded`）用來決定數值與行為。
  - [ ] 把現有 `createEnemy` 改為從資料表 + 隨機調整，而不是完全硬編。

- [ ] **關卡 / 層數系統**
  - [ ] 在 `state` 中新增 `stage`（目前層數）、`stageProgress`（本層已打幾隻 / 目標）。
  - [ ] 設計每一層的目標：例如「一般怪 3 隻 + Boss 1 隻」。
  - [ ] 根據 `stage` 對敵人 HP / ATK 做成長倍率（難度曲線）。
  - [ ] 完成一層時給一次性獎勵（例：金幣、進階元素或特殊召喚石），並解鎖下一層。

- [ ] **強敵機制調整（整合到關卡）**
  - [ ] 讓「強敵」機率可隨層數略微提升或由特定關卡保證出現。
  - [ ] 強敵在 UI 上增加明顯標記（顏色 / 特殊邊框）。
  - [ ] 強敵掉落表與普通怪分開設定（更高的裝備與進階元素機率）。

---

### 二、戰鬥決策變多（多種行動）

- [ ] **行動列表設計**
  - [ ] 在 UI 中將原本單一「攻擊」按鈕改為多個行動按鈕列。
  - [ ] 初步行動：`普攻`、`重擊`、`防禦`、（視情況）`治療`。
  - [ ] 定義每種行動的效果（傷害倍率、是否消耗 HP / 回合、是否影響敵人下回合攻擊）。

- [ ] **戰鬥計算與狀態效果**
  - [ ] 在 `state` 中支援玩家與敵人的戰鬥狀態（例如：`stunned`, `bleeding`, `shield`）。
  - [ ] 擴充戰鬥流程：  
        玩家行動 → 套用效果 / 狀態 → 檢查敵人死亡 → 敵人若存活則執行其行動 → 套用效果 / 狀態 → 檢查玩家死亡。
  - [ ] 為狀態設計簡單數值：暈眩（敵人本回合不能反擊）、流血（每回合扣固定傷害）、護盾（先扣護盾再扣 HP）。
  - [ ] 在戰鬥日誌中清楚顯示每步驟內容（行動、傷害、狀態變化）。

- [ ] **UI 與操作體驗**
  - [ ] 保持按鈕數量在 3〜4 個以內，避免新手過載。
  - [ ] 針對新加入的行動，在第一次進入戰鬥時顯示簡短說明（可用一行提示文字）。

---

### 三、裝備與屬性成長

- [ ] **角色屬性擴充**
  - [ ] 在 `state` 中加入：`def`（防禦）、`critChance`（暴擊率）、`critMult`（暴擊傷害倍率）。
  - [ ] 更新戰鬥傷害公式：考慮攻擊、防禦、暴擊（但保持公式簡單可調）。
  - [ ] 在 UI 中僅顯示最關鍵幾個屬性（例如：攻擊、防禦、生命），避免過多數字。

- [ ] **裝備欄位與裝備資料**
  - [ ] 在 `state` 中新增 `equipmentSlots`（例如：`weapon`, `armor`, `accessory`）。
  - [ ] 定義裝備資料結構：`slot`, `name`, `affix`, `power`, `stats`（影響的屬性）。
  - [ ] 建立「穿戴 / 脫下」邏輯，讓玩家可以在背包中選擇要穿哪件裝備。
  - [ ] UI 中標示目前已裝備的物品與其加成。

- [ ] **裝備強化 / 升級**
  - [ ] 設計簡單的強化系統：使用金錢或召喚素材讓裝備 +1、+2···。
  - [ ] 每次強化增加固定或比例的 `power` / 相關屬性。
  - [ ] 強化 UI：在背包或交易所旁新增「強化」入口，不與賣出混淆。

---

### 四、技術與重構注意事項

- [ ] **保持單檔架構但做適度區塊化**
  - [ ] 在 `app.js` 內用註解區分模組區塊（敵人 / 關卡 / 戰鬥 / 裝備），避免邏輯混在一起。
  - [ ] 盡量讓每個功能使用小函式（例如：`applyDamage`, `applyStatus`, `nextStage`），方便之後再抽離成模組。

- [ ] **測試擴充**
  - [ ] 更新 `TEST-MATRIX.md`：加入新戰鬥行動、裝備屬性、關卡機制的測試條目。
  - [ ] 擴充 `test.js`：至少驗證新增屬性存在、基礎計算公式正確、主要流程不報錯。
